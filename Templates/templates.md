# Obsidian Vault — Templates Reference

> This file documents the template system for this vault: what each template is for,
> when to use it, and how the whole system is set up. Keep this in `/Templates/`.

---

## 1. Installation (One-Time Setup)

### Step 1 — Enable the Templates Core Plugin
1. Open Obsidian → click ⚙️ **Settings**
2. Go to **Core plugins**
3. Find **Templates** → toggle **ON**
4. Close Settings

### Step 2 — Point Templates to the Right Folder
1. Open ⚙️ **Settings** → **Templates** (now visible in the left sidebar under Plugin Options)
2. Set **Template folder location** → `Templates`
3. Set **Date format** → `YYYY-MM-DD`
4. Set **Time format** → `HH:mm`
5. Close Settings

### Step 3 — Wire Up Daily Notes
1. Open ⚙️ **Settings** → **Daily Notes**
2. Set **New file location** → `04 Journal`
3. Set **Date format** → `YYYY-MM-DD`
4. Set **Template file location** → `Templates/Daily Note`
5. Close Settings

### Step 4 — Verify It Works
1. Press `Ctrl/Cmd + P` → type **Open today's daily note** → press Enter
2. A new note should open in `/04 Journal/` pre-filled with the Daily Note template
3. Inside any other new note, press `Ctrl/Cmd + P` → type **Insert template** → select a template from the list

> ✅ Setup complete. Templates are now available vault-wide.

## 2. How to Use Templates Day-to-Day

### Inserting a template manually
1. Create a new note anywhere in the vault
2. Give it a title (important — this becomes the node name in your graph)
3. Press `Ctrl/Cmd + P`
4. Type: `Insert template`
5. Select the appropriate template from the list
6. Fill in or delete the placeholder fields

### Daily note (automatic)
- Press `Ctrl/Cmd + P` → **Open today's daily note**
- Or click the calendar icon in the left sidebar (if Calendar plugin is installed)
- Template is inserted automatically — no manual step needed

## 3. Template Inventory

### 3a. Daily Note
**File:** `Templates/Daily Note.md`
**Auto-applied:** Yes — via Daily Notes plugin
**Lives in:** `/04 Journal/YYYY-MM-DD.md`

**Purpose:**
Lightweight daily capture. Not a journal — a net for catching ideas, observations,
and fragments across all three domains before they disappear.
Use it every day, even if you only write one bullet point.

**Key sections:**
| Section | What goes here |
|---|---|
| 🔗 Context | Auto-links to your 3 hub notes — leave as-is |
| 🧠 Quick Capture | Anything, no structure required |
| 🔧 Embedded / Projects | ESP32 work, project thoughts |
| 🥑 Nutrition / Health | Meals, experiments, observations |
| 💰 Finance / Thinking | Market thoughts, trade ideas |
| 🔗 Potential Links | Concepts worth linking later |
| 📌 Worth Keeping | Fragments that might become a real note |

**Behavioral rule:** Don't try to complete every section. Use what's relevant that day, ignore the rest.

---

### 3b. Project Hub Note
**File:** `Templates/Project Hub.md`
**Auto-applied:** No — insert manually
**Lives in:** `/02 Projects/<Project Name>/<Project Name>.md`

**Purpose:**
The master note for any active project. One per project.
Everything related to the project links back here.
Claude Code uses this as the anchor when working on a project with you.

**Key sections:**
| Section | What goes here |
|---|---|
| Goal | One sentence — what it does and why |
| Components / Stack | Links to sensors, libraries, concepts used |
| Tasks | Checklist of next actions |
| Log | Dated progress entries, append-only |
| Open Questions | Things unresolved — never delete these prematurely |
| Related Knowledge | Links to Knowledge notes generated from this project |

**When to create:** At the start of any new project, before writing anything else.
**Claude Code trigger:** *"Let's start a new project"* or *"New project: [name]"*

---

### 3c. Knowledge Note
**File:** `Templates/Knowledge Note.md`
**Auto-applied:** No — insert manually or let Claude Code generate it
**Lives in:** `/01 Wiki/<Concept Name>.md`

**Purpose:**
A reusable, stable note about a single concept.
The test: *"Would this note be useful after the project that generated it is done?"*
If yes → Knowledge note. If no → leave it as a Working Note.

**Key sections:**
| Section | What goes here |
|---|---|
| Summary | 1–2 sentence plain-language definition |
| Detail | Mechanism, nuance, how it actually works |
| Relevance to My Work | Why *you* care — not generic, personal |
| Related | Links to related concepts and projects |
| Open Questions | Things still unclear — worth keeping visible |
| Sources | URLs, datasheets, references |

**Naming rule:** Always use canonical nouns. `DS18B20`, not `temp sensor notes`.
**Source field:** Set to `claude` if generated by Claude Code, `manual` if written by you.
**When to create:** When a concept appears in more than one project, or when something
feels stable and reusable enough to stand on its own.

---

### 3d. Working Note
**File:** `Templates/Working Note.md`
**Auto-applied:** No — insert manually
**Lives in:** `/02 Projects/<Project Name>/` (alongside the hub note)

**Purpose:**
Raw, in-progress, session-level notes. Wiring logs, debugging sessions,
experiment results, build diary entries. Not meant to be permanent or clean.
Think of it as your scratchpad for a specific project session.

**Key sections:**
| Section | What goes here |
|---|---|
| Context | What you're trying to do right now |
| Session Log | Dated entries, append as you work |
| Observations | Things you noticed |
| Problems / Errors | Error messages, failures, unexpected behavior |
| Next Steps | What to do next time you open this |
| Extractions | Concepts worth promoting to a Knowledge note later |

**Behavioral rule:** Never clean these up obsessively. Their value is in the raw detail.
The **Extractions** section is the only one that feeds forward — Claude Code scans it during maintenance passes.

---

### 3e. Journal Note
**File:** `Templates/Journal Note.md`
**Auto-applied:** No — insert manually
**Lives in:** `/04 Journal/<Descriptive Title>.md`

**Purpose:**
Reflection on something worth thinking through — a project post-mortem,
a trading decision, a design choice, a trip, an insight.
Unlike the Daily Note (which is a capture net), a Journal Note is a lens:
you focus it on one thing and think out loud.

**Key sections:**
| Section | What goes here |
|---|---|
| What happened / What's on my mind | Situation or event — write in prose |
| What I think about it | Your actual reaction, not a summary |
| What it connects to | A few intentional links — not automatic |
| What I want to remember | 1–2 things worth keeping |
| What's still unresolved | Open threads — may feed into a project or future note |

**Naming rule:** Name by topic, not date.

The `created` date in frontmatter handles the temporal metadata.

---

### 3f. Entity
**File:** `Templates/Entity.md`
**Auto-applied:** No — insert manually or let Claude Code generate it
**Lives in:** `/01 Wiki/<Entity Name>.md`

**Purpose:**
A named thing: a person, company, product, technology, location, instrument.
Use when you need to track something concrete rather than an abstract concept.

**Key sections:**
| Section | What goes here |
|---|---|
| What it is | One-line identity |
| Key facts | Bullet list of stable facts |
| Relevance to My Work | Why you track this specifically |
| Related | Links to concepts and other entities |
| Sources | Source summary pages |

**Naming rule:** Canonical proper nouns. `DS18B20`, `NVIDIA`, `RRSP` — not `temp sensor` or `the nvidia thing`.

---

### 3g. Source Summary
**File:** `Templates/Source Summary.md`
**Auto-applied:** No — created by Claude Code during ingest
**Lives in:** `/01 Wiki/Source: <Original Title>.md`

**Purpose:**
One note per ingested source. The permanent record of what a source said and what was extracted from it.
Created automatically during the ingest workflow — you rarely fill this in by hand.

**Key sections:**
| Section | What goes here |
|---|---|
| Citation | Full attribution |
| Summary | 3-5 sentences — what the source argues |
| Key claims | Each claim linked to the concept page it fed |
| Extracted into | Every concept/entity page created or enriched |
| Quotes worth keeping | One short verbatim quote max |
| Open questions raised | Things the source left unresolved |

**Naming rule:** Always `Source: <Original Title>` — the prefix is load-bearing for Dataview queries.

---

### 3h. Comparison
**File:** `Templates/Comparison.md`
**Auto-applied:** No — created by Claude Code during query fileback
**Lives in:** `/01 Wiki/Comparison: <Topic>.md` or `/01 Wiki/<A> vs <B>.md`

**Purpose:**
Cross-source synthesis: where sources agree, disagree, or fill gaps on a specific question.
Created when a query produces insight worth preserving, not just answering.

**Key sections:**
| Section | What goes here |
|---|---|
| Question | What is being compared and why |
| Points of agreement | Where sources align |
| Points of disagreement | Source-attributed divergences |
| Synthesis | Working position or open question |
| Related | Concepts and entities this comparison informs |

## 4. Template Decision Tree

```
I want to write something new...
│
├── It's about today in general → Daily Note
│
├── It's about a specific project
│   ├── Starting the project → Project Hub Note
│   └── Working session / log / experiment → Working Note
│
├── It's a concept I'll reuse across projects → Knowledge Note (type: concept)
│
├── It's a named thing (person, product, technology) → Entity
│
├── It's a summary of something I just read/watched → Source Summary
│
├── It's comparing two or more sources on a question → Comparison
│
└── It's a reflection on something that happened → Journal Note
```

## 5. How Templates Feed the Graph

```
Daily Note ──────────────────────────→ Hub Notes (auto, via Context line)
     │
     └── "Worth Keeping" ──────────→ Knowledge Note (promoted manually)

Project Hub ──────────────────────→ Knowledge Notes (linked in Related)
     │
     └── Working Notes ────────────→ Knowledge Notes (via Extractions section)

Journal Note ─────────────────────→ Project Hub or Knowledge Note (intentional links)
```

## 6. Frontmatter Reference

All notes use consistent frontmatter so Dataview can query them later.

```yaml
---
type: daily | project | knowledge | working | journal
domain: embedded | nutrition | finance | general      # omit on daily/journal
status: active | paused | done                        # projects only
tags: []
created: YYYY-MM-DD
source: manual | claude | web                         # who created it
---
```

**The `source` field matters:** Once you have enough notes, you can run a Dataview query
to find all Claude-generated knowledge notes and do a review pass — validating,
correcting, or enriching them based on what you've learned since.
